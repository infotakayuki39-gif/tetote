## コーディング規約

### 基本

- **BEM**: `.block__element--modifier`
- **FLOCSS プレフィックス**: `l-`（layout）/ `c-`（component）/ `p-`（project）/ `u-`（utility）
- `!important` 禁止（外部ライブラリ上書きの最終手段を除く）
- `id` へのスタイル付与禁止
- 新しいスタイルは対応する FLOCSS レイヤーのファイルに書く

### CSS 設計原則

CSS は「予測・再利用・保守・拡張」の4原則を満たすこと。

- 特定ページ・セクションに依存しない**ブロック（コンポーネント）単位**で構築する
- `id` セレクタは絶対に使わない。すべてクラスで管理する

### タイポグラフィ

- **letter-spacing**: 原則書かない。デザインカンプで明示的に指定されている場合のみ設定する

### レイアウトとプロパティ

- **幅**: `width: 100%` + `max-width` を併用し、横漏れを防ぐ。固定幅は避ける
- **高さ**: `height` 固定禁止。`padding` または `min-height` でコンテンツに応じた高さを確保する
- **横並び**: `float` 禁止。Flexbox / Grid を使う
- **position: absolute**: 最終手段。高さ消失による崩れに細心の注意を払う
- **overflow: hidden**: レイアウト不備の隠蔽に使わない。幅の設定ミスなど根本原因を解決する
- **余白**: 「親→子」方向に設定し、子が外側を押し出す指定は避ける
- **ボタン・タグ等の可変テキスト要素**: 左右に十分な `padding` を設け、テキストが枠に密着しないようにする

### HTML 構造

- セマンティックなマークアップを徹底する（`section`, `article`, `nav`, `main` 等）
- `section` / `article` の内部には必ず見出し（`h2`〜`h6`）を含める
- タグの親子関係（入れ子ルール）を厳守し、W3C バリデーションエラーを出さない
- `img` や `a` は直接置かず、`div` 等のブロック要素で囲んでメンテナンス性を確保する

### レスポンシブ

- すべての画面幅で表示崩れが起きないよう考慮する

1. `npm run build` が通ること
2. ブラウザでスマホ表示・PC 表示を目視確認（特にブレイクポイント前後）
